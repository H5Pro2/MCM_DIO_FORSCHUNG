## Abgleich-Regel

Ein alter Mechanikpunkt wird nur übernommen, wenn er in die neue Schichtung
passt:

```text
Welt
-> Sinneswahrnehmung
-> MCM-Feld
-> Memory
-> Thought
-> Realitätsprüfung
-> Handlung
-> Konsequenz
-> Regulation
```

Nicht übernommen werden:

- harte innere Gates außer dem Value-Gate
- Impuls als Entry-Anker
- Hypothese als Realität
- Strategie als Vorgabe
- Debug als Mechanik
- doppelte Regler mit gleicher Wirkung auf denselben Zielbereich

## Nur als Übergang oder Archiv

Diese alten Begriffe dürfen im Code noch als Kompatibilitätsfelder auftauchen,
aber nicht als aktive Fachsprache des Bauplans:

- `strategy_*`
- `motor_*`
- `action_*`
- `block`
- `gate` außerhalb des Value-Gates
- `permission` als innere Freigabelogik
- `impulse_preferred`
- Hypothesenfamilie als reale Formfamilie

Zielzustand:

```text
Strategie -> emergente tragende Reaktion
Motorik -> Handlung / Beteiligungsnähe
Action -> Handlung oder Konsequenz
Block -> Abstand, Vorsicht, Reorganisation oder Nicht-Handlung
Permission -> Realitätsprüfung / tragende Nähe
Impulse -> Wahrnehmungsdruck
```

## Nicht aktiv übernehmen

Diese alten Muster sind fachlich verworfen:

- Hypothese erzeugt direkt Entry.
- Impuls setzt Entry.
- Innere Unsicherheit blockiert hart.
- Positive Spannung bedeutet automatisch Ruhe.
- Energie bedeutet automatisch Stress.
- Debug-Frequenz beweist Mechanik.
- Viele Regler wirken parallel auf denselben Zielwert.
- DIO lernt Strategie als festes Regelwerk.

## Offene Prüfspur

Aus dem alten Mechanikbestand bleiben nur diese Prüfaufgaben aktiv:

1. Core-Daten weiter reduzieren: Welt, Sehen, MCM-Feld.
2. Alte `strategy_*`, `motor_*`, `action_*` im Code zuordnen und abbauen.
3. Hypothesenpfad konsequent von realer Form trennen.
4. Denk-Last als Feld- und Regulationslast führen, nicht als Endlossuche.
5. Entry nur aus tragender Nähe und gültiger Order-Geometrie bilden.
6. Debug in getrennte Beobachtungsschichten aufteilen.
